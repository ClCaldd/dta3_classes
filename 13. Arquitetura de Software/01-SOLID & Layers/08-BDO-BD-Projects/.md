No BD, é comum evitarmos o uso de ORMs automáticos como o Entity Framework em favor de classes DAL (Data Access Layer) próprias. Isso acontece porque, em sistemas de alta escala, o controle sobre o banco de dados é um ativo estratégico.

1. Por que não usar o EF?
O Entity Framework foca em produtividade, gerando o SQL automaticamente para o desenvolvedor. Já a abordagem de DAL manual foca em controle. Ao escrevermos nossas próprias consultas e mapeamentos, garantimos que o banco de dados execute exatamente o que planejamos, sem "queries fantasmas" ou processamento desnecessário de memória.

2. Como isso impacta o projeto?
Performance Cirúrgica: Conseguimos otimizar cada SELECT para trazer apenas o que é essencial, utilizando índices e recursos específicos do SQL Server que o EF muitas vezes ignora.

Mapeamento Customizado: Se a tabela no banco tem nomes de colunas complexos ou antigos, a nossa DAL faz a "limpeza" desses dados antes de entregá-los para o resto do sistema.

Independência Tecnológica: Ao não acoplarmos o sistema às facilidades do EF, criamos uma estrutura onde o Banco de Dados é um serviço especializado, e não apenas uma extensão do código C#.

3. O papel da Arquitetura
Trabalhar sem EF exige mais código (o famoso boilerplate), mas é aqui que a Arquitetura de Camadas brilha. Como nossas interfaces estão na Application, o restante do software não sabe (e não importa) se os dados vêm via EF, Dapper ou SQL Puro. Isso permite que o BDO troque a tecnologia de acesso a dados sem quebrar uma única regra de negócio.
