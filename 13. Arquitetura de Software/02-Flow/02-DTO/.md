# Data Transfer Object (DTO)

DTO, Objeto de transporte de dados, são utilizados para passar pacote 
de dados de uma parte da aplicação para outra, ou então para resposta



### record X class
Principais diferenças:
- **Imutabilidade:** Records incentivam imutabilidade, diferente de classe
- **Valor X Referência:** Como viram anteriormente, objetos carregam referência a endereços da memória, portanto um objeto só é igual a outro quando ambos apontam para o mesmo endereço de memória, já no caso do record ele compara o valor de cada atributo dentro dele.
- **Criação:** Para criarmos records que não precisam ser objetos complexos, podemos inicia-los de maneira muito mais simples e limpa:

Classe
```cs
var u = new Usuario(id, nome);
public class Usuario(int id, string nome)
{
    public int Id { get; } = id;
    public string Nome { get; set; } = nome;
}
```

Record
```cs
var u = new Usuario(id, nome);
public record Usuario(int Id, string Nome);
```
O compilador faz tudo sozinho. Ele já cria as propriedades Id e Nome, define como init (imutável) e faz a atribuição.
Mesmo que pareça besteira, imagine quando temos mais que 5 atributos na classe? Teremos uma classe suja e grande sem necessidade

---
- **Cópia com alteração:** Records são imutáveis, portanto se quiser um outro objeto que tenha tudo igual mas um atributo diferente, em C# temos o operador 'with' para resolver
```cs
var obj1 = new Usuario(1, "Gru");
var obj2 = obj1 with { Nome = "Minion" };
```
---
- Embora não seja de todo modo útil recorrentemente, um record possui um ToString() automático, montando em formato JSON no terminal
- **Parâmetros:**
Record: Os parâmetros do construtor viram Propriedades Públicas.
Classe: Os parâmetros do construtor primário não viram propriedades. Eles funcionam como campos privados que você só consegue usar dentro da classe. Se você quiser expor o id para fora da classe, você precisa declarar uma propriedade explicitamente.


### Transferência de dados
É comum utilizarmos objetos do tipo record para a transferencia de objetos dentro da aplicação, isso se deve aos pontos citados a cima.
Evite retornar no controller a sua entidade completa, isso envolve diversos problemas de perfomance, segurança.

Exemplo:

Considere que temos a entidade 'User', com o seguinte modelo:
```cs
public class User

    int Id {get;set;}

    File Picture {get;set;}
    string Password {get;set;}

    Gender Gender {get;set;}
    string LastName {get;set;}
    string CPF {get;set;}
    string Mother {get;set;}
```

Esse é um dos pontos de atenção, mas esse tipo de falha pode acontecer mesmo 
montando um DTO, digamos que para rconomizar tempo, decido reaproveitar o DTO
que é usado na tela de admin ao visualizar um usuário específico para outras, 
como por exemplo essa da listagem.
Portanto é de extrema importância saber a responsabilidade de cada DTO, montar
a resposta correta para aquela requisição.

Vamos piorar o cenário e pensar no caso onde um usuário possui um usuário que é
seu chefe:

```cs
public class User
{
  int Id {get;set;}
  string Name {get;set;}
  string Edv {get;set;}
  DateOnly Birth {get;set;}

  User Boss {get;set;}
  int BossId {get;set;}
}
```

Em sistemas que trabalham com serialização automática de objetos, retornar entidades 
completas pode gerar estruturas extremamente grandes e difíceis de controlar.
No exemplo acima, ao retornar um funcionário, o sistema também poderia retornar o seu 
chefe, que por sua vez possui outro chefe, e assim sucessivamente, criando uma árvore 
gigantesca de dados.

Isso aumenta:
- o consumo de memória
- o tamanho da resposta
- o tempo de processamento
- o tráfego da rede
